# OpenClaw 精简版调研方案

## 1. 调研目标

基于 `openclaw/openclaw` 现有实现，输出一个可落地的精简版本方案，满足以下约束：

1. 只支持 Telegram 渠道。
2. 只支持 OpenAI 模型，并支持 Codex 登录方式（OAuth + 可选复用本机 Codex 登录态）。
3. 工具能力只保留 Mac 文件操作、浏览器操作（可选保留 `system.run`，仅限 macOS）。
4. 支持斜杠命令 `/coding_agent`，用于调用 Codex。
5. 提供精简控制台（本地 CLI/TUI）。

## 2. OpenClaw 源码映射（与需求直接相关）

### 2.1 网关与运行时骨架

- 启动入口：`src/index.ts`、`src/entry.ts`
- 网关主流程：`src/gateway/server.impl.ts`
- 渠道生命周期管理：`src/gateway/server-channels.ts`

结论：OpenClaw 的主架构是“统一网关 + 多渠道插件 + 统一 agent runtime”。精简版不需要完整插件系统，但可复用“单渠道启动 + 统一消息分发”的骨架思想。

### 2.2 Telegram 链路

- 监控与轮询/回调：`src/telegram/monitor.ts`
- Bot 创建与中间件：`src/telegram/bot.ts`
- 事件处理：`src/telegram/bot-handlers.ts`
- 消息处理器组装：`src/telegram/bot-message.ts`
- 上下文构建：`src/telegram/bot-message-context.ts`
- 回复派发：`src/telegram/bot-message-dispatch.ts`
- 原生命令注册：`src/telegram/bot-native-commands.ts`

结论：Telegram 已有完整实现，精简版建议保留“monitor + bot + handler + dispatch”主链路，移除多账号/复杂群组策略/高级事件（如 reaction、复杂迁移逻辑）。

### 2.3 命令与回复分发

- 回复主入口：`src/auto-reply/reply/get-reply.ts`
- agent 执行：`src/auto-reply/reply/agent-runner.ts`
- 命令分发中心：`src/auto-reply/reply/commands-core.ts`
- ACP 命令：`src/auto-reply/reply/commands-acp.ts`
- 命令注册：`src/auto-reply/commands-registry.ts`

结论：`/acp spawn codex` 已是现成功能。`/coding_agent` 最省成本方案是做 alias，内部转译到 ACP spawn。

### 2.4 OpenAI / Codex 登录与模型

- OpenAI 文档：`docs/providers/openai.md`
- Codex OAuth：`src/commands/openai-codex-oauth.ts`
- OAuth 流程：`src/commands/oauth-flow.ts`
- OpenAI 鉴权应用：`src/commands/auth-choice.apply.openai.ts`
- OpenAI-Codex 默认模型：`src/commands/openai-codex-model-default.ts`
- 模型归一化：`src/agents/model-selection.ts`
- 鉴权读取：`src/agents/model-auth.ts`
- 本机 Codex 凭证兼容实现（可选）：`src/agents/cli-credentials.ts`

结论：OpenClaw 对 OpenAI API Key 与 OpenAI-Codex OAuth 都有实现，且有现成模型映射逻辑。精简版可以只保留 `openai-codex/*` 与少量 `openai/*` 兼容。

### 2.5 工具层（文件/浏览器/Mac）

- 工具总装：`src/agents/pi-tools.ts`
- OpenClaw 扩展工具：`src/agents/openclaw-tools.ts`
- 浏览器工具：`src/agents/tools/browser-tool.ts`
- 节点/系统运行工具：`src/agents/tools/nodes-tool.ts`

结论：OpenClaw 默认工具集合很大。精简版应改为 allowlist，仅暴露：

- 文件操作：`read` / `write` / `edit` / `apply_patch`（按需）
- 浏览器操作：`browser.*`
- 可选：`system.run`（仅 macOS，默认关闭）

### 2.6 控制台

- CLI 命令：`src/cli/tui-cli.ts`
- 网关聊天客户端：`src/tui/gateway-chat.ts`
- TUI 主实现：`src/tui/tui.ts`

结论：当前 TUI 功能较重。精简版可只保留“会话选择 + 输入 + 流式输出 + slash 命令输入”。

## 3. 精简边界（做/不做）

### 3.1 本次必须保留

1. Telegram 单渠道收发消息。
2. OpenAI（重点 openai-codex OAuth）模型调用。
3. 文件与浏览器工具能力。
4. `/coding_agent` slash 命令。
5. 轻量控制台。

### 3.2 本次明确移除

1. Discord/Slack/Web 等其他渠道。
2. 多 provider（Anthropic、Google、xAI 等）。
3. 大部分扩展工具（tts/image/web-search/cron/canvas/subagents 全量控制面）。
4. 复杂权限系统、跨渠道线程绑定、插件市场能力。

## 4. 推荐实现策略

### 方案 A（推荐）：基于 OpenClaw 架构“抽骨架重组”

- 复用成熟链路：Telegram、OpenAI-Codex OAuth、agent runner、browser 工具。
- 新建 `lite` 目录聚合最小功能，不直接在原模块上大删大改。

优点：

- 风险最低，能复用已验证行为。
- 后续如要加回能力，可平滑接回主架构。

缺点：

- 初始代码仍有一定“继承复杂度”。

### 方案 B：从零写最小代理框架

优点：

- 代码最干净、最小。

缺点：

- 需要重写 OAuth、Telegram 边界处理、流式回复等，重复造轮子。

结论：选方案 A。

## 5. 关键技术风险与处理

1. Codex 登录态兼容风险：不同机器上 `~/.codex/auth.json` 与 keychain 可用性不一致。  
   处理：主路径用 OAuth；兼容读取本地登录态作为 fallback。
2. 浏览器工具环境风险：browser host/sandbox 依赖运行环境。  
   处理：默认走本地可执行路径，启动前做健康检查。
3. Telegram 消息并发与去重风险：重复 update、分片消息。  
   处理：沿用 `update dedupe + sequentialize + debounce` 思路。
4. `/coding_agent` 命令体验风险：如果直接暴露 ACP 参数，用户使用成本高。  
   处理：提供简化参数封装（如仅支持 `--cwd`、`--mode`）。

## 6. MVP 验收标准

1. Telegram 中发送普通消息，机器人能用 `openai-codex/gpt-5.3-codex` 正常回复。
2. 执行 `/coding_agent` 能创建并调用 codex 会话，返回执行结果。
3. Agent 可调用文件读写/编辑工具。
4. Agent 可调用浏览器导航/截图等核心动作。
5. 本地控制台可连接同一网关会话，支持消息交互与 `/coding_agent`。

## 7. 建议里程碑

1. 里程碑 1：跑通 Telegram + OpenAI-Codex OAuth + 基础对话。
2. 里程碑 2：加入工具白名单（文件/浏览器）并完成安全限制。
3. 里程碑 3：实现 `/coding_agent` alias（映射 ACP spawn）。
4. 里程碑 4：上线精简控制台并联调。
5. 里程碑 5：补最小测试与发布脚本。
