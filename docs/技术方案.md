# OpenClaw 精简版技术方案

## 1. 方案目标

构建一个 `coke-claw-lite`，仅覆盖以下能力：

1. Telegram 单渠道。
2. OpenAI 模型（重点 `openai-codex` OAuth 登录）。
3. 工具仅保留 Mac 文件操作、浏览器操作（可选 `system.run`）。
4. 支持 `/coding_agent` slash 命令调用 Codex。
5. 提供精简控制台。

## 2. 总体架构

```text
Telegram Update
   -> telegram-adapter-lite
   -> router-lite (slash / normal)
   -> agent-runtime-lite (openai/openai-codex)
   -> tools-lite (fs + browser [+mac run])
   -> response-stream
   -> Telegram Reply

Console-lite
   -> same router-lite / same runtime / same session store
```

设计原则：

1. 单一运行时：只留一个默认 agent。
2. 能力白名单：默认拒绝一切非必要工具。
3. 命令最小化：只保留 `/help`、`/coding_agent`、`/model`（可选）。
4. 配置最小化：无多账号、多渠道、多插件。

## 3. 目录建议

```text
src/
  lite/
    app.ts                     # 启动器
    config.ts                  # 精简配置加载与校验

    channels/
      telegram/
        monitor.ts             # 轮询/webhook（复用 openclaw monitor 逻辑）
        bot.ts                 # bot + middleware + dedupe + 顺序化
        handler.ts             # update -> message/slash 事件

    runtime/
      router.ts                # 普通消息 / slash 命令分发
      reply.ts                 # 流式回复管线
      slash/
        help.ts
        coding-agent.ts        # /coding_agent -> acp spawn

    auth/
      openai-codex-oauth.ts    # OAuth 登录主路径
      codex-cli-bridge.ts      # 可选：读取 ~/.codex/auth.json + keychain

    tools/
      index.ts                 # tools allowlist 组装
      fs-tools.ts              # 文件操作封装
      browser-tools.ts         # 浏览器操作封装
      mac-run.ts               # 可选 system.run

    acp/
      spawn-codex.ts           # 统一封装 ACP spawn

    console/
      cli.ts                   # 精简控制台入口
      chat.ts                  # 发送/接收/渲染
```

## 4. 配置设计（最小）

```yaml
app:
  name: coke-claw-lite

telegram:
  botToken: ${TELEGRAM_BOT_TOKEN}
  mode: polling # or webhook

model:
  provider: openai-codex
  primary: openai-codex/gpt-5.3-codex

auth:
  mode: oauth # oauth | codex-cli

tools:
  fs: true
  browser: true
  macRun: false

commands:
  native: true
  codingAgent:
    defaultMode: persistent
    defaultThread: auto
```

## 5. 核心流程

### 5.1 启动流程

1. 加载配置与环境变量。
2. 初始化 auth（优先 OAuth，失败时可回退 codex-cli-bridge）。
3. 初始化 runtime（固定 provider 白名单：`openai`、`openai-codex`）。
4. 初始化 tools allowlist（`fs + browser + optional macRun`）。
5. 启动 Telegram monitor。
6. 启动 console-lite（可选，单独命令）。

### 5.2 消息处理流程

1. Telegram 收到 update。
2. 去重 + 顺序化。
3. 解析是否 slash：
   - 是：进入 `slash router`。
   - 否：进入 `reply pipeline`。
4. reply pipeline 调模型并流式返回。
5. 结束后写入会话存储（最小字段）。

### 5.3 `/coding_agent` 流程

命令格式建议：

```text
/coding_agent <task...> [--cwd /path] [--mode persistent|oneshot]
```

执行策略：

1. 解析参数，构造 ACP spawn 请求。
2. 固定 `agentId=codex`，`runtime=acp`。
3. 默认 `thread=auto`，从当前 Telegram 会话绑定上下文。
4. 调用 `acp/spawn-codex.ts`。
5. 将创建结果与首轮执行输出回传 Telegram。

等效映射（内部）：

```text
/acp spawn codex --mode <mode> --thread auto --cwd <cwd>
```

## 6. 关键复用点（来自 openclaw）

1. Telegram 适配：复用 `monitor.ts + bot.ts` 设计，保留 dedupe 和顺序化。
2. OpenAI-Codex OAuth：复用 `openai-codex-oauth.ts` + `oauth-flow.ts`。
3. 模型归一化：复用 `model-selection.ts` 中 `openai -> openai-codex` 归一化规则。
4. 工具装配：参考 `pi-tools.ts`，改成硬编码 allowlist。
5. ACP 调度：复用 `commands-acp`/`sessions_spawn(runtime=acp)` 机制。
6. 控制台：复用 `gateway-chat.ts` 通讯方式，去掉复杂 overlay/status 功能。

## 7. 裁剪清单

### 7.1 直接不引入

1. 多渠道（Discord/Slack/Web）。
2. 多 provider/onboard 全流程。
3. 插件系统（tools/commands/channels 动态插件）。
4. 非必要工具：tts/image/web-search/cron/canvas/subagent 全套。
5. 复杂安全/审计策略（保留最小 deny-by-default 即可）。

### 7.2 保留但简化

1. 会话系统：只保留本地 KV + 当前渠道 sessionKey。
2. slash 系统：保留最小 registry（2-3 个命令）。
3. TUI：简化为单屏输入输出。

## 8. 安全策略（精简版）

1. 默认 deny 全部工具，只显式放行白名单。
2. 文件操作限制在 workspace（禁止访问系统敏感目录）。
3. 浏览器工具限制允许域名（可配置 allowlist）。
4. `system.run` 默认关闭，仅在 macOS + 显式配置开启。
5. `/coding_agent` 的 `cwd` 做路径规范化和越权检查。

## 9. 最小接口定义

### 9.1 Slash Command Handler

```ts
export interface SlashHandler {
  name: string;
  run(input: {
    text: string;
    sessionKey: string;
    from: { id: string; username?: string };
    channel: "telegram" | "console";
  }): Promise<{ text: string }>;
}
```

### 9.2 ACP Spawn Service

```ts
export interface SpawnCodexParams {
  task: string;
  cwd?: string;
  mode?: "persistent" | "oneshot";
  thread?: "auto" | "off";
  sessionKey: string;
}
```

## 10. 实施计划

1. Phase 1: 骨架搭建
- 完成 `lite/app.ts`、配置加载、Telegram 最小收发。

2. Phase 2: 模型与认证
- 接入 OpenAI-Codex OAuth；实现 `openai-codex` 默认模型。

3. Phase 3: 工具白名单
- 接入 `fs`、`browser`，验证回归；按需加 `macRun`。

4. Phase 4: `/coding_agent`
- 完成命令解析与 ACP spawn 映射，支持 Telegram + console。

5. Phase 5: 控制台
- 完成 console-lite 与网关统一会话联动。

6. Phase 6: 测试与交付
- 补端到端 smoke 测试：消息回复、slash、工具调用、登录。

## 11. 验收用例（建议）

1. Telegram 普通对话：返回流式回复。
2. `/coding_agent 写一个 hello world`：成功创建 codex 会话并返回结果。
3. 文件工具：创建文件、编辑文件、读取文件。
4. 浏览器工具：打开页面、抓取标题、截图。
5. 控制台：同一 sessionKey 可继续对话。
